## Known Pain Points

**The fork still breathes gstack DNA.** The codebase setup, directory names (`~/.gstack/`),
binary names (`gstack-update-check`, `gstack-config`), package.json name (`"gstack"`),
analytics paths, and skill preambles all reference gstack rather than chief. Working on
chief features means constantly stepping around scaffolding that isn't yours.

The core problem: chief is a fork with a thin identity layer on top, not yet its own thing.
Every `/chief` session reads preamble output mentioning gstack. Every `~/.gstack/` write
reinforces the wrong brand. This is friction on every PR.

## Engineering Conventions

From `CLAUDE.md`:

- **Always bisect commits.** Each commit is one logical change — rename/move separate from
  behavior changes, template changes separate from generated file regeneration.
- **SKILL.md files are generated.** Edit `.tmpl` files, run `bun run gen:skill-docs`, commit both.
- **bun test before every commit** (free, <2s). `bun run test:evals` before shipping (paid, diff-based).
- **CHANGELOG.md is for users**, not contributors. Lead with what the user can now DO.
- **Platform-agnostic skills** — never hardcode framework-specific commands. Read CLAUDE.md, ask if missing, persist the answer.
- **Skills must work without CLAUDE.md.** AskUserQuestion as fallback, then persist to CLAUDE.md.

Key commands:
```
bun test               # free tests — run before every commit
bun run test:evals     # paid evals — run before shipping (~$4/run max)
bun run gen:skill-docs # regenerate SKILL.md files from templates
bun run build          # gen docs + compile binaries
bun run skill:check    # health dashboard for all skills
```

All in — suggest skills whenever relevant: `/review`, `/qa`, `/ship`, `/investigate`, `/retro`, etc.

### Coaching Focus Areas
Chief will pay attention to these during code reviews and coaching:
- **Chief identity vs gstack bleed** — flag any new code that writes to `~/.gstack/`, references
  `gstack-*` binaries, or uses gstack branding where chief should own it
- **Skill template quality** — are changes to `.tmpl` files matched by a `gen:skill-docs` run?
  Generated files should never be edited directly.
- **Test coverage for new skills** — chief-specific skills (chief-init, chief-push, chief-cook)
  lack E2E evals. Flag opportunities to add coverage.
- **Commit bisection** — this project has strict atomic commit rules; flag kitchen-sink commits
